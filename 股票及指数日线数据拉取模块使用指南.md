# 股票日线数据同步模块使用指南

> Sequoia-X V2 数据同步框架 — 基于 baostock 的 A 股日线数据全量/增量自动同步

---

## 一、模块概述

### 1.1 设计目标

将 baostock A 股日线数据自动同步到本地 SQLite 数据库，为选股策略和 LLM 分析提供持久化的行情数据。

### 1.2 架构位置

```
sequoia_x/
├── data/
│   ├── engine.py          # 查询层 — 仅负责数据读取和基础股票池过滤
│   └── sync.py            # 同步层 — DataSync 类，全部同步逻辑
└── main.py                # 入口 — 命令行分发
```

**职责分离**：
- `DataEngine`（engine.py）：纯查询，提供 `get_ohlcv` / `get_base_stock_pool` 等接口供选股策略调用
- `DataSync`（sync.py）：纯同步，管理 baostock 连接生命周期、增量拉取、缺失修复、历史回填

### 1.3 核心特性

| 特性 | 说明 |
|------|------|
| 交易日自动识别 | 调用 baostock `query_trade_dates` API 精确判断，非交易日自动跳过 |
| 增量同步 | 仅拉取每只股票最新日期到今日之间的缺失数据 |
| 上市/退市检测 | 每次同步对比 `query_stock_basic` 全量列表与本地数据，自动增删 |
| 断连恢复 | 连续 50 次 API 错误自动重连，指数退避重试（最多 2 轮） |
| 幂等写入 | `INSERT OR REPLACE` 基于 `UNIQUE(symbol, date)` 约束，中断可安全重跑 |
| 数据完整性检查 | `check_missing()` 诊断缺失数据，`repair_missing()` 自动补填 |
| 进度日志 | 每 100 只股票输出进度，同步完成记录 sync_log 表（含覆盖率、耗时） |
| 微信推送 | 同步完成/失败通过 WxPusher 推送到微信 |

## 二、数据存储

### 2.1 数据库结构

数据库文件：`data/sequoia_v2.db`

**stock_daily 表**（日线行情数据）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 自增主键 |
| symbol | TEXT | 6 位股票代码（如 "600000"） |
| date | TEXT | 交易日期（YYYY-MM-DD） |
| open | REAL | 开盘价（前复权） |
| high | REAL | 最高价（前复权） |
| low | REAL | 最低价（前复权） |
| close | REAL | 收盘价（前复权） |
| volume | REAL | 成交量（股） |
| turnover | REAL | 换手率（%） |
| **UNIQUE** | (symbol, date) | 联合唯一约束 |

**stock_list 表**（股票列表）：

| 字段 | 类型 | 说明 |
|------|------|------|
| symbol | TEXT PRIMARY KEY | 6 位股票代码 |
| listed_date | TEXT | 上市日期 |
| delisted_date | TEXT | 退市日期（NULL=在册） |
| updated_at | TEXT | 最后更新时间 |

**sync_log 表**（同步日志）：

| 字段 | 类型 | 说明 |
|------|------|------|
| date | TEXT UNIQUE | 同步日期 |
| status | TEXT | ok / error |
| stock_count | INTEGER | 股票总数 |
| delisted_count | INTEGER | 退市股票数 |
| new_listed_count | INTEGER | 新上市股票数 |
| backfilled_days | INTEGER | 补填天数 |
| is_trade_day | INTEGER | 是否为交易日 (0/1) |
| api_status | TEXT | baostock 连接状态 |
| coverage | REAL | 最新日数据覆盖率 |
| duration_seconds | REAL | 同步耗时（秒） |
| error_msg | TEXT | 错误信息 |

## 三、DataSync 类方法参考

### 3.1 核心同步方法

#### `sync_daily(force=False) → dict`
增量日线同步，拉取每只股票的最新日线数据。

```python
# 日常自动模式（含交易日判断 + 时间门控 17:30）
result = sync_mgr.sync_daily(force=False)

# 手动强制模式（忽略交易日和时间门控）
result = sync_mgr.sync_daily(force=True)
```

**返回示例**：
```json
{"status": "ok", "stock_count": 4800, "is_trade_day": true, "error": ""}
```

**内部机制**：
1. 交易日检查（非交易日→ `status: "skipped"`）
2. 时间门控（< 17:30 → 跳过，baostock 日线入库时间）
3. 逐只拉取增量（last_date + 1 → today）
4. 请求间隔 50ms 防封
5. 连续 50 次错误自动重连 baostock

#### `run_full() → dict`
完整三阶段同步管线：

```bash
python main.py --sync-only    # 等同 run_full()
```

```
Phase 1: sync_stock_list()  → 股票列表同步（上市/退市检测）
Phase 2: sync_daily(force=False) → 增量日线同步
Phase 3: repair_missing(days=5) → 缺失补填
```

#### `backfill() → dict`
历史数据全量回填（从 `settings.start_date` 到今天所有交易日）。

```bash
python main.py --backfill
```

**耗时参考**：~5200 只股票，约 800 个交易日，预计 45-60 分钟。

### 3.2 诊断与修复方法

#### `check_missing(days=5) → dict`
诊断最近 N 天的数据完整性。

```python
report = sync_mgr.check_missing(days=5)
# {"latest_date": "2026-06-09", "trade_days_expected": 4, 
#  "total_missing": 340, "missing_by_symbol": {...}}
```

#### `repair_missing(days=5, max_stocks=None) → dict`
修复缺失数据。

```bash
# 仅修复前 50 只（默认快速模式）
python main.py --repair

# 修复全部缺失股票
python main.py --repair --all
```

**修复策略**：
1. 调用 `check_missing()` 诊断缺失
2. 按缺失数降序修复（优先补最重要的）
3. 失败股票自动重试 2 轮（指数退避 0.5s/1.0s）

### 3.3 辅助方法

#### `is_trade_day(check_date=None) → bool`
精确判断交易日。

```python
sync_mgr.is_trade_day()                    # 今天
sync_mgr.is_trade_day(date(2026, 6, 9))    # 指定日期
```

采用 fail-open 策略：baostock 异常时假定为交易日，避免漏同步。

#### `get_active_stocks() → dict`
获取全量 A 股列表并检测变化。

```python
{"symbols": ["600000", ...], "new_listed": ["688xxx"], "delisted": ["000xxx"], "count": 5206}
```

#### `sync_stock_list() → dict`
持久化股票列表变化到 stock_list 表。

## 四、命令行使用指南

### 4.1 命令速查

| 命令 | 用途 | 适用场景 |
|------|------|----------|
| `python main.py --sync-only` | 日常自动同步 | cron 17:45 定时执行 |
| `python main.py --repair` | 快速修复前 50 只缺失 | 部分数据异常 |
| `python main.py --repair --all` | 全量修复所有缺失 | 手动补数据、应急恢复 |
| `python main.py --backfill` | 首次全量历史回填 | 初次部署或数据清空后 |
| `python main.py --skip-llm` | 仅选股不分析 | 快速选股验证 |
| `python main.py` | 完整选股流程 | cron 20:55 定时执行 |

### 4.2 典型使用场景

**场景 1：首次部署数据初始化**
```bash
python main.py --backfill    # 回填全部历史数据（45-60 分钟）
```

**场景 2：每日自动更新**
```cron
# crontab 配置
10 19 * * 1-5 cd /path/to/sequoia-x && /path/to/python main.py --sync-only >> logs/sync.log 2>&1
55 20 * * 1-5 cd /path/to/sequoia-x && /path/to/python main.py >> logs/daily.log 2>&1
```

**场景 3：发现数据缺失的手动修复**
```bash
# 快速修复（前 50 只，< 1 分钟）
python main.py --repair

# 全量修复（全部缺失股票，约总股票数 × 0.5秒）
python main.py --repair --all
```

**场景 4：在选股前确认数据完整性**
```python
from sequoia_x.core.config import get_settings
from sequoia_x.data.sync import DataSync

sync_mgr = DataSync(get_settings())
report = sync_mgr.check_missing(days=5)
# 检查 coverage，决定是否先 repair 再选股
```

### 4.3 预期耗时

| 操作 | 数据量 | 50ms 间隔 | 说明 |
|------|--------|-----------|------|
| `sync_daily(force=False)` | 仅当日增量 | 2-3 分钟 | 仅拉取 1 天数据 |
| `sync_daily(force=True)` | 5200 只 × 1 天 | **~40 分钟** | 全量增量 |
| `repair_missing(days=5)` | 视缺失量 | 每只 ~0.5 秒 | 与缺失股票数成正比 |
| `backfill()` | 5200 只 × 800 天 | **~50 分钟** | 仅首次需要 |
| `run_full()` | 正常一天 | 5-8 分钟 | 日常自动同步 |

## 五、从旧代码借鉴的设计

### 5.1 借鉴 "03 代码" 的股票代码清洗

原 `03_stock_code_data_20251115.py` 每 30 天从 baostock 拉取全量标的物 → 过滤指数 → 保留 A 股代码 → 存 CSV。

**改进后的方案**：
- 不再用 CSV 文件，改用 SQLite `stock_list` 表
- 不再定时 30 天拉取，改为**每次同步实时对比** `query_stock_basic`
- 过滤逻辑升级：仅保留沪深 A 股（`sh.6 / sz.0 / sz.3`）
- 自动检测 `new_listed`（远程有/本地无）和 `delisted`（本地有/远程无）

### 5.2 借鉴 "04 代码" 的交易更新

原 `04_stock_index_daily_baostock_update_20251219.py` 每日 18:00 拉取→对比 CSV 最后日期→判断是否需要更新→逐股写入 CSV。

**改进后的方案**：
- 不再对比文件修改日期，改用 **baostock `query_trade_dates` API** 精确判断
- 不再存 CSV（僵死方案），改用 SQLite（支持原子写入、并发查询）
- 支持 `force=True` 手动强制模式（原代码的 `RUN_TIME=1` 手动模式等价）
- 连接复用 + 断连重试（原代码无此机制）
- 自动多日补填（原代码仅支持"差一日补一日"或"差多日补多日"）

### 5.3 原 04 代码发现的问题及修复

| 原问题 | 新方案修复 |
|--------|-----------|
| 用文件修改日期判断交易日（不准确） | 用 `query_trade_dates` API |
| stock_basic_baostock.csv 偶尔为空 | 用 `query_stock_basic` + SQLite，失败时重试 2 轮 |
| 无断连恢复机制 | 连续 50 次错误自动重连 + 指数退避 |
| 无数据完整性检查 | `check_missing()` + `repair_missing()` 组合 |
| 多进程并行导致 baostock 死锁 | 严格单进程 + 50ms 间隔 |

## 七、常见问题

### Q1: 运行时出现 "Broken pipe" 或 "10002007" 错误
**原因**：baostock 高频调用触发限流，或连接长时间未活动被服务端断开。

**应对**：代码已内置自动重连机制（连续 50 次错误后重连 baostock），以及指数退避重试。通常重连后恢复。

### Q2: 同步后数据库只有部分股票有数据
**原因**（历史 Bug，已修复）：
1. ~~`_write_to_db` 使用 `DELETE WHERE date = ?` 导致逐只写入时后者删前者~~ → 已改为 `INSERT OR REPLACE`
2. ~~请求无间隔导致 baostock 断连~~ → 已添加 50ms 间隔

**解决方法**：运行 `python main.py --repair --all` 追加缺失数据。

### Q3: 非交易日运行同步会怎样？
`sync_daily(force=False)` 检测到非交易日会直接返回 `{"status": "skipped"}`，不调用 baostock API。如需强制运行（如怀疑交易日判断有误），使用 `force=True` 或 `--repair --all`。

### Q4: 如何确认同步是否成功？
1. 查看微信推送（同步完成后自动推送摘要）
2. 查看 `sync_log` 表：`SELECT * FROM sync_log ORDER BY date DESC LIMIT 1;`
3. 查看日志文件：`logs/sync_YYYYMMDD.log`

### Q5: 015/016 项目如何切换到本数据库？
在 015/016 项目中添加以下数据读取逻辑（替代原有的 CSV 读取）：
```python
import sqlite3, pandas as pd
conn = sqlite3.connect("data/sequoia_v2.db")
df = pd.read_sql("SELECT * FROM stock_daily WHERE symbol=?", conn, params=("600000",))
```

### Q6: 同步速度怎么优化？
- 当前固定 50ms 间隔是为了防止 baostock 限流
- 如确认网络稳定，可改为 20ms（修改 `sync.py` 中 `time.sleep(0.05)` 为 `time.sleep(0.02)`)
- 注意：间隔过小（< 10ms）可能导致断连

### Q7: 数据库损坏怎么办？
1. 备份当前数据库：`cp data/sequoia_v2.db data/sequoia_v2.db.bak`
2. 删除并重建：`rm data/sequoia_v2.db`
3. 重新回填：`python main.py --backfill`
